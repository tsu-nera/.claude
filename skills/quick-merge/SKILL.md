# Quick Merge - 手元の変更を一気通貫でmergeまで

手元の未コミット変更を、ブランチ作成→型チェック→commit→push→PR作成→auto-merge
まで人間の確認なしで実行する**薄いオーケストレーター**。
中身は既存コマンド（`/pr` → `/lgtm`）の連結であり、ロジックを重複実装しない。

Issueは不要。その場の簡単な修正を一気通貫でmergeしたい場合に使う。
Issue起点なら `/issue-to-merge` を使うこと（こちらはその非Issue版）。

## 使い方
`/quick-merge`

## いつ使うか

**使うべき**:
- 手元に未コミットの簡単な変更がある（修正・小機能・リファクタ）
- 既存パターンを踏襲し、設計判断が不要
- 自信があり、PRレビューを挟まず自動マージしてよい

**使うべきでない**:
- 設計判断・複数アプローチの検討が必要 → `/pr` でPR作成に留め人間判断を仰ぐ
- Issue起点で進めたい → `/issue-to-merge` を使う
- main直コミットが許される変更（`resources/`, `logs/weekly/`）→ このスキルは使わず直コミット運用に従う
  （理由: コード変更はPR経由が原則。ログ文言もエージェント解析前提のコードでありPR必須）

## Instructions for Claude:

### Phase 0: 適性判定

```bash
git status
```

- 未コミット変更が無い → 「対象の変更がない」と報告して停止
- 変更が広範・設計判断を伴うと判断したら停止し、`/pr` 単体（PR作成のみ）の使用を提案
  （このスキルは「迷いなく自動マージしてよい時だけ使う」前提。迷ったら停止）

### Phase 1: PR作成

`/pr` を Skill ツールで起動する。

- `/pr` はブランチ作成（mainにいる場合）→型チェック→commit→push→PR作成までを行う
- ステージングは関連ファイルのみ明示的に行うこと（`resources/` をblanketで含めない）
- `gh pr create` 本文末尾にグローバルCLAUDE.md規定のセッションIDフッターを付与すること
- 型チェック失敗 → 修正してから再実行。修正不能なら中断してユーザーに報告（mergeしない）
- PR作成成功 → PR番号を取得して Phase 2 へ

### Phase 2: マージ

`/lgtm <PR番号>` を Skill ツールで起動する。

- CIが回っている場合、failしていればマージしない（lgtm の指示に従う）
- worktree環境の後処理も lgtm のロジックに委譲する

### Phase 3: 完了報告

merge済みのcommit hashとPR番号、実施フェーズのサマリをユーザーに報告。

## エラー時の対応

- **Phase 0で複雑と判定**: `/pr` 単体の使用を提案して終了
- **Phase 1で型チェック失敗・修正不能**: 中断してユーザーに報告。PRは作らない
- **Phase 2でCI失敗等**: lgtm の指示に従い中断。PRは残したままユーザー判断を待つ

## スコープ外

- 設計判断・人間レビューを挟みたい → `/pr` を直接使う
- Issue起点 → `/issue-to-merge` を使う
- main直コミット運用（`resources/`, `logs/weekly/`）→ このスキルの対象外
