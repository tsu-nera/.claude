# Test PR - PR動作確認ワークフロー

PRのTest planに沿って動作確認を実行し、失敗時は修正→再テストのループを最大3回回す。
全パスしたら結果をPRコメントに投稿する。

## 使い方
`/test-pr <PR番号>`

## Instructions for Claude:

**Orchestrator-Workerパターン。** 自身（sonnet）がオーケストレーターとして判断し、テスト実行と修正はサブエージェントに委任する。

---

### Phase 1: PR情報の収集とブランチ準備

```bash
gh pr view <PR番号> --json title,body,files,headRefName
```

PRのbodyから以下を抽出:
- **Test plan**: リスト項目を実行対象として取得
- **Closes #xx**: 関連Issue番号（結果記録用）

**Test planが空の場合**: テスト項目なしとしてPRコメントに「動作確認項目なし、CIパスのみで判断」と投稿して終了。

**ブランチ準備**:
1. PRのブランチ名を取得（headRefName）
2. 既存worktreeがそのブランチを使用しているか確認: `git worktree list`
3. 既存worktreeがあれば → そのworktreeディレクトリで実行
4. なければ → `gh pr checkout <PR番号>` で切り替え

worktreeの場合は `pnpm install` を実行（node_modulesが共有されないため）。

---

### Phase 2: テスト実行（testerサブエージェント）

テスト項目ごとにsonnetサブエージェント（tester）を起動する。

**並列化の判断**（オーケストレーターが行う）:
- 独立した項目（異なるコマンド、相互依存なし）→ 並列起動
- 依存のある項目（実行順序が重要）→ 逐次起動
- 判断に迷ったら逐次

```
Agent(
  model: "sonnet",
  prompt: "以下のテスト項目を実行し、結果を返せ。コード修正はしない。
    テスト項目: <項目>
    実行コマンド: <コマンド>
    期待結果: <期待結果>
    作業ディレクトリ: <worktreeパスまたはリポジトリルート>

    実行後、以下の形式で結果を返せ:
    - 結果: PASS / FAIL
    - 実行コマンド: （実際に実行したコマンド）
    - 出力: （stdout/stderr）
    - 失敗理由: （FAILの場合のみ）"
)
```

**実行ルール**:
- サーバ接続が必要な項目（「要サーバ接続」記載あり）→ ユーザーに実行環境を確認してから実行
- 破壊的操作（データ変更、送金等）→ **絶対に実行しない。** dry-runオプションがあれば使う
- タイムアウトは適切に設定（デフォルト120秒、長いものは明示的に延長）

---

### Phase 3: 結果判定（オーケストレーター）

全testerの結果を集約し判定する。

- **全パス** → Phase 5へ
- **失敗あり** → Phase 4へ（修正ループ、残り回数を確認）
- **3回修正しても失敗** → Phase 5へ（失敗状態で結果記録）

---

### Phase 4: 修正ループ（coderサブエージェント）

失敗したテスト項目について、sonnetサブエージェント（coder）を起動してコード修正する。

```
Agent(
  model: "sonnet",
  prompt: "以下のテスト失敗を修正せよ。
    失敗したテスト項目: <項目>
    エラーログ: <testerの出力>
    対象ファイル: <PRの変更ファイル一覧>
    作業ディレクトリ: <worktreeパスまたはリポジトリルート>

    修正後:
    1. pnpm run check で型チェックを通す
    2. git add <変更ファイル> && git commit で修正をコミット
    3. git push"
)
```

**修正後**: Phase 2に戻り、**全テスト項目**を再実行（修正が他の項目を壊していないか確認）。

**ループ上限**: 3回。超えたらPhase 5で失敗として記録。

---

### Phase 5: 結果の記録

テスト結果をPRコメントとして投稿する。

```bash
gh pr comment <PR番号> --body "$(cat <<'EOF'
## 🧪 動作確認結果

### テスト環境
- 実行日時: YYYY-MM-DD HH:MM
- 実行環境: local / conoha1 / conoha2
- ブランチ: <headRefName>

### テスト項目と結果

| # | テスト項目 | 結果 | 備考 |
|---|----------|------|------|
| 1 | ... | ✅ PASS | |
| 2 | ... | ❌ FAIL | エラー内容 |

### 修正ループ
- 修正回数: N/3
- 修正内容: <coderが行った修正のサマリ>

### 実行ログ
<details>
<summary>テスト1: ...</summary>

```
$ コマンド
出力
```
</details>

### 総合判定
✅ 全テストパス / ❌ 失敗あり（修正上限到達）

🤖 Tested with [Claude Code](https://claude.com/claude-code)
EOF
)"
```

---

### Phase 6: 報告

テスト結果のサマリをユーザーに報告する。

- **全パス（修正なし）**: 「全テストパス。PRコメントに結果を投稿しました。」
- **全パス（修正あり）**: 「N回の修正後に全テストパス。修正コミットをプッシュ済み。」
- **失敗（上限到達）**: 失敗内容と修正試行の経緯を報告。人間の判断を求める。

---

## スコープ外

- **テスト項目の設計**: `/issue-to-pr` が担当済み
- **マージ判断**: 人間が行う
- **破壊的操作**: 送金、データ削除等は絶対に実行しない
